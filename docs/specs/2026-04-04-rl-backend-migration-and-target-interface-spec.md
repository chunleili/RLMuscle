# RL 后端迁移与目标接口细则

日期：2026-04-04
关联总纲：`docs/plans/2026-04-04-rl-onboarding-to-motion-tracking.md`

## 目标

定义 RL 从 `simpleArm` 迁移到更复杂 backend 的边界，以及为动作数据库和视频骨架预留的统一目标接口要求。

这份细则解决两个问题：

1. 什么叫“同一 RL 任务在不同 backend 上迁移”。
2. 后续更复杂目标输入应该如何接入，而不破坏已有环境层。

## 迁移范围

backend 迁移顺序固定为：

1. `simplearm_mj`
2. `simplearm_xpbd`
3. `couple3_bicep`

不允许跳过前一层直接进入下一层，否则训练问题将难以定位。

## backend 迁移原则

### 必须保持一致的内容

以下语义在迁移时应尽量保持一致：

- action 仍表示 excitation
- 核心任务仍表示姿态控制或轨迹跟踪
- 主要 reward 结构不变
- 核心评估指标不变

### 允许调整的内容

以下内容允许随 backend 变化做有限调整：

- reward 权重
- control rate
- 最大 episode 长度
- 数值稳定相关超参数

### 不应随意变化的内容

以下内容不应因为 backend 变复杂就被重新定义：

- 任务本质
- policy 输出语义
- 成功判据的大方向
- 核心日志字段

否则跨 backend 对比将失去意义。

## XPBD simpleArm 的角色

XPBD simpleArm 不是最终任务，而是中间桥梁。  
它的作用是回答一个关键问题：

在保持单关节任务不变的情况下，从轻量 MuJoCo backend 切到 volumetric / XPBD backend 后，RL 是否仍然能训练。

因此 XPBD simpleArm 的目标不是追求新任务，而是做迁移验证。

## couple3 第一版边界

`couple3` 第一版 RL 任务严格限定为：

- 固定目标肘角
- 短参考肘角轨迹
- 单肌主导控制

以下内容暂不进入第一版：

- 多肌协同分配
- 高自由度动作
- 长时复杂动作追踪
- 全身任务

这样做的目的是把 `couple3` 作为“真实几何验证”而不是“所有复杂性一次上线”。

## 统一目标接口

为了支持后续动作数据库与视频骨架输入，RL 层需要面向统一的 target abstraction，而不是直接绑定某个数据源。

## 目标来源分类

后续目标输入分成两类：

### 1. motion target

指干净、明确、可直接用于控制的参考目标，例如：

- 常值角度
- 程序生成轨迹
- 动作数据库中的关节角曲线

### 2. inverse target

指从视频骨架等上游估计得到的目标，特点是：

- 可能有噪声
- 可能缺帧
- 可能不完全符合物理约束

这两类来源都应通过同一个 target interface 暴露给环境层，但在训练与评估中不应混为一谈。

## target interface 输出要求

统一输出应至少包含：

- 当前时刻目标姿态
- 可选未来窗口参考
- 元数据：`fps`、`horizon`、`joint_names`

如果后续扩展到多关节，也应沿用同一结构，而不是为每种数据源单独设计环境接口。

## 与单关节阶段的关系

尽管当前阶段只做单关节任务，这份接口约束仍然需要提前确定。  
原因是：

- `simpleArm` 上验证的是环境逻辑
- 后续数据源扩展不应迫使重写 RL 主框架
- 单关节接口设计不应阻碍多关节扩展

## 迁移阶段的验收要求

### simplearm_mj -> simplearm_xpbd

至少验证以下问题：

- 相同任务能否继续训练
- reward 变化趋势是否一致
- 关键控制链路量是否仍可观测

### simplearm_xpbd -> couple3_bicep

至少验证以下问题：

- 固定目标肘角任务是否能建立闭环
- 训练是否仍然可复现
- 是否出现新的明显数值不稳定或 reward 失真

## 风险点

### 1. backend 差异过大导致接口名义一致、语义不一致

例如 observation 字段虽然名字相同，但实际数值含义、采样频率或归一化方式不同。  
这会使“迁移成功”看起来成立，实际上无法比较。

### 2. 过早把 noisy target 接入训练主线

视频骨架属于更高层输入问题，如果过早接入，会掩盖 RL 控制是否真的成立。

### 3. 在 `couple3` 中过早引入多肌问题

多肌冗余会显著增加控制难度，应晚于单肌主导的肘关节验证。

## 输出物

本阶段应形成：

- backend adapter 的一致性约束
- target interface 的统一字段定义
- `couple3` 第一版 RL 任务边界说明
- 一份迁移成功与失败的判定标准
