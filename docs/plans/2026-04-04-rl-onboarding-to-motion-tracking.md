# RL 入门到动作追踪实施总纲

日期：2026-04-04

## 背景

当前 physics 部分已经进入可用于控制验证的阶段，下一步需要引入 RL，验证“激活值控制肌肉并完成目标动作”的主链路是否能稳定工作。  
这一阶段不直接追求全身复杂动作，也不直接追求从视频反演激活，而是先建立一个可训练、可验证、可迁移的 RL 工作流。

## 总目标

建立一条分阶段推进的 RL 路线，使项目能够从最简单的单关节任务出发，逐步过渡到真实肌肉几何上的动作追踪，并最终为动作数据库与视频骨架驱动预留统一接口。

这条路线要解决的不是单个 demo，而是以下三个问题：

1. RL 是否能够稳定控制现有肌骨耦合系统。
2. RL 环境接口、奖励设计和评估方式是否足够清晰，能够支持后续扩展。
3. 从简单后端到复杂后端的迁移路径是否顺畅，避免在一开始把 RL 问题和 physics 问题混在一起。

## 总体路线

当前总路线固定为四个阶段：

1. `simpleArm + 固定目标姿态`
2. `simpleArm + 短参考轨迹`
3. `XPBD simpleArm / couple3 + 同类任务迁移`
4. `动作数据库 / 视频骨架目标接口 + 多关节扩展`

四个阶段不是并行推进，而是逐级推进。只有前一阶段通过验证，后一阶段才有意义。

## 核心决策

### 1. 不直接从 `couple3` 起步

`couple3` 更接近最终任务，但它同时引入了更复杂的几何、数值稳定性和训练成本。  
如果第一版 RL 就从 `couple3` 开始，会很难区分以下问题到底来自哪里：

- 环境接口设计错误
- reward 定义不合理
- policy 学不会
- 后端本身太难训练

因此，第一阶段必须从 `simpleArm` 起步，把 RL 闭环先跑通。

### 2. RL 层与 physics 层分离

RL 入口统一放在 `src/rl/`，而不是放进 `src/VMuscle/rl/`。  
`VMuscle` 继续负责 physics backend、solver 和 muscle coupling，`rl` 负责环境、目标、奖励、训练与评估组织。

这条边界是为了保证后续扩展时：

- RL 逻辑不被具体 solver 绑死
- backend 替换时不重写整个训练框架
- 动作数据库与视频骨架目标可以直接复用环境层

### 3. 第一版优先验证控制链路，不优先追求复杂生物力学

这一阶段的重点是验证如下控制链路是否在 RL 框架下可用：

`excitation -> activation -> force/torque -> joint angle`

第一版的成功标准是“能训练、能控制、能复现、能分析”，不是“最高保真”或“最复杂任务”。

### 4. 最小激活正则预留但不强推

从任务定义上看，姿态追踪可能存在“达到同一姿态但用力明显过大”的情况，因此最小激活正则很可能会需要。  
但在第一版中，它只作为可调 reward 选项预留，不作为过强默认约束。是否提升其权重，放到阶段性实验后再决定。

## 阶段划分

### 阶段 1：simpleArm 固定目标姿态

目标：

- 建立第一个标准 RL 环境
- 完成单关节、单目标姿态控制
- 验证基础 reward 和 episode 定义
- 建立最小训练与评估流程

交付：

- 一份针对 `simpleArm pose` 的细则 spec
- 一套可复现的训练与评估方案
- 一组基础成功判据

### 阶段 2：simpleArm 短参考轨迹

目标：

- 从静态姿态控制升级到时间相关轨迹跟踪
- 明确 tracking 任务的 observation、target 和 metrics
- 为动作数据库接口打基础

交付：

- 一份针对 `simpleArm track` 的细则 spec
- 一组标准轨迹任务类型
- 一套 tracking 评估指标

### 阶段 3：后端迁移到 XPBD simpleArm 和 couple3

目标：

- 尽量保持 RL 接口语义不变
- 将已验证任务迁移到更复杂后端
- 区分训练难度来自 RL 设计还是来自 physics backend

交付：

- 一份针对后端迁移的细则 spec
- 一套跨 backend 的一致性检查要求
- `couple3` 的第一版任务边界

### 阶段 4：统一目标接口与多关节扩展

目标：

- 为动作数据库和视频骨架目标预留统一 target interface
- 让单关节任务的环境层可扩展到多关节
- 为最终全身任务建立稳定输入接口

交付：

- 目标接口约束
- 多关节扩展的边界定义
- 数据源分层策略

## 文档拆分原则

本总纲只回答以下问题：

- 为什么这样分阶段
- 每一阶段解决什么问题
- 哪些设计决策已经固定
- 哪些风险需要后续 spec 专门处理

以下内容不再放在总纲中，而进入 `docs/specs/`：

- 具体 observation 定义
- 具体 reward 公式和初始权重
- episode 终止条件
- success threshold
- backend adapter 的输入输出约束
- target provider 的具体字段
- 训练和评估的脚本组织方式

## 主要风险

### 1. 奖励定义与真实控制目标不一致

如果 reward 只优化角度误差，可能会学出不合理的高激活或高频抖动策略。  
这需要通过日志和对比实验来判断是否引入更强的激活正则。

### 2. backend 迁移后任务语义漂移

如果迁移到 XPBD 或 `couple3` 后，观测、控制频率或状态定义发生明显变化，就会破坏“同任务跨 backend 对比”的意义。  
因此后续 spec 必须约束什么可以调整，什么不应变化。

### 3. 过早引入复杂目标来源

动作数据库和视频骨架本质上属于更后期的输入源问题。  
如果第一阶段就把其耦合进来，会模糊 RL 基础任务是否真的成立。
